## Vision

### 10x Check

Current plan: Extract two libraries from existing code. Ship `go get` + `npm install`.

10x version: Not just libraries — a **developer experience platform** for local AI. The library doesn't just expose inference — it provides streaming, context management, custom tool registration, named model selection, progress feedback, structured output, and actionable error messages. A developer adds local AI to their existing app in 4 lines of code, with tool calling, context injection, and streaming — all on-machine, zero cloud dependencies.

The difference: "here's a library that wraps llama-server" vs "here's the way developers add AI to apps." small-llm becomes the sqlite of local LLM — you embed it, it just works.

### Platonic Ideal

A developer types `go get github.com/muthuishere/small-llm/go`. Then:

```go
ai, _ := llm.New()
ai.AddContext("Here is our product catalog: ...")
ai.RegisterTool("search", mySearchFunc)
answer, _ := ai.Ask(ctx, "Find the cheapest laptop")
```

Four lines. The library downloads the model (with progress), starts inference, reads context, invokes the tool, returns a structured answer. All on-machine. The developer feels: "Wait, that's all? This is like adding a database — one import and it works."

For React:
```jsx
const { ask } = useWebLLM({ context: productCatalog, tools: { search: searchFn } });
```
Same magic. Browser-only. Two lines of setup.

### Key Insight (from user)

"I don't even need it to know the capital of France. I want tool calling — it can connect to API, I can give some context and then it can answer. We don't need a big model. The position is: user can build small-level React components with context, tool calling, and a little search, both from a backend and frontend."

**The model is not the product. The plumbing is the product.** small-llm is a lightweight reasoning engine for tool dispatch, context extraction, and structured tasks. Not local ChatGPT — local AI plumbing.

## Scope Decisions

| # | Proposal | Effort | Decision | Reasoning |
|---|----------|--------|----------|-----------|
| 1 | Streaming inference | S | ACCEPTED | Table stakes for any chat UX — both backends already support it |
| 2 | Context management API | S | ACCEPTED | Core to the vision — "give context, get answers" is the primary use case |
| 3 | Custom tool registration | S | ACCEPTED | Without this, tools are a demo. With this, tools are a platform |
| 4 | Named model constants (models.Qwen2_5_0_5B) | XS | ACCEPTED | Developer experience — no GGUF paths in user code |
| 5 | Go HTTP middleware | S | DEFERRED | Good but not essential for library launch |
| 6 | Progress callbacks for model download | XS | ACCEPTED | First-run experience — 400MB silent download is unacceptable |
| 7 | Structured output via JSON schema | S | ACCEPTED | Constrained decoding makes tool calling reliable |
| 8 | Descriptive error logging | XS | ACCEPTED | Actionable errors with fix suggestions, not opaque failures |

## Accepted Scope (added to this plan)

- **Streaming inference** — `ai.AskStream()` in Go, `{ askStream, tokens }` in React hooks
- **Context management** — `ai.AddContext()` / `ai.SetContext()` in Go, `<LLMProvider context={}>` in React
- **Custom tool registration** — `ai.RegisterTool(name, desc, handler)` in Go, `useTools({ custom })` in React
- **Named model constants** — `models.Qwen2_5_0_5B`, `models.Qwen2_5_3B` + `llm.WithModelURL()` for custom
- **Progress callbacks** — `llm.WithProgress(func)` in Go, `{ downloadProgress }` in React
- **Structured output** — `ai.AskStructured(ctx, q, schema)` using llama-server constrained decoding
- **Descriptive error logging** — every error includes what happened + what to do + relevant paths

## Deferred to TODOS.md

## Updated API Surface (post-expansion)

### Go Library

```go
// Core
ai, _ := llm.New()                                    // default model, auto-bootstrap
ai, _ := llm.New(llm.WithModel(models.Qwen2_5_0_5B))  // named model (explicit size)
ai, _ := llm.New(llm.WithModel(models.Qwen2_5_3B))    // larger model
ai, _ := llm.New(llm.WithModelURL("https://huggingface.co/..."))  // custom GGUF
ai, _ := llm.New(llm.WithProgress(func(d, t int64){})) // download progress

// Inference
answer, err := ai.Ask(ctx, "question")                 // batch response
err := ai.AskStream(ctx, "question", func(token string){}) // streaming (SSE under the hood)
var result MyStruct
err := ai.AskStructured(ctx, "question", &result)      // constrained JSON via json.RawMessage schema

// Context — two modes:
ai.AddContext("anonymous context text...")              // appends to prompt (unnamed, stackable)
ai.SetContext("catalog", catalogText)                   // named context (upsert by key)
ai.RemoveContext("catalog")                             // remove named context

// Tools — handler signature: func(ctx context.Context, input string) (string, error)
ai.RegisterTool("search", "Search products by query", searchFn)
answer, _ := ai.Ask(ctx, "find cheapest laptop")       // auto-invokes registered tools

// Interface for testability
type Completer interface {
    Complete(messages []ChatMessage, maxTokens int, temp float64) (*ChatResponse, error)
    CompleteStream(messages []ChatMessage, maxTokens int, temp float64, cb func(string)) error
}

// Lifecycle
defer ai.Close()
```

### React Library

```jsx
// Hooks
const { ask, askStream, tokens, isLoading, error, downloadProgress } = useWebLLM()
const { ask, askStream, isLoading, error } = useLLM()       // server mode
const { messages, send, clear, isLoading } = useChat()       // conversation state (replaces Zustand store in library consumers)
const { call, result, availableTools } = useTools({ search: { description: "Search", handler: searchFn } })

// Provider — model constants use explicit names matching HuggingFace
<LLMProvider model={models.Qwen2_5_0_5B} context={catalogText}>
  <App />
</LLMProvider>

// Browser models map to MLC model IDs internally:
// models.Qwen2_5_0_5B → "Qwen2.5-0.5B-Instruct-q4f32_1-MLC"
```

## CEO Review Findings (Sections 1-10)

### Architecture Decisions (Section 1)
- **Completer interface** for testability and multi-backend future
- **Functional options** (`WithModel()`, `WithProgress()`) replace config package coupling
- **Constrained JSON decoding** for tool calling (replaces regex parsing)
- **Concurrency**: Go queues requests (mutex), React rejects if busy (check `isLoading`)

### Error & Rescue Map (Section 2)
4 silent failure gaps identified — all in inference path:
- ErrInference (500 from llama-server → panic)
- ErrEmptyResponse (empty string → silent)
- ErrStreamInterrupt (SSE drop → partial response)
- ErrUnknownTool (nonexistent tool → ignored)
All fixed as part of typed errors work (Expansion #8).

### Security (Section 3)
Minimal attack surface (local library, no network endpoints). Tool execution is developer-owned — documented, not sandboxed.

### Data Flow (Section 4)
All shadow paths mapped. Concurrent inference handled: Go queues, React rejects.

### Code Quality (Section 5)
One anti-pattern: `downloader.go` string-based JSON parsing → fix during extraction.

### Tests (Section 6)
13 flows mapped. Completer interface enables full unit testing without llama-server. Integration tests for real model loading (nightly CI). React tests via Jest+RTL with mocked engines.

### Performance (Section 7)
No issues. Single-user local library. Model memory managed by llama-server.

### Observability (Section 8)
Go: `log/slog` (stdlib, redirectable). React: `console.warn/error`.

### Deployment (Section 9)
Library publish: `git tag go/v0.1.0` + `npm publish @small-llm/react`. Users pin versions.

### Long-Term (Section 10)
Reversibility: 4/5. Completer interface enables Phase 2 (multi-backend), Phase 3 (model registry), Phase 4 (orchestration).

### Design/UX (Section 11)
SKIPPED — library has no UI.
